TEST 1: LOAD TEST (5 minutes, ramp to 50 users/sec, hold)
=====================================================================

RESULTS:
  Total requests:       12,060
  OK:                   12,060  (100% success!)
  KO:                        0  (0% failure)
  Mean response time:     367 ms
  Min:                    320 ms
  Max:                  1,858 ms
  p50:                    359 ms
  p75:                    369 ms
  p95:                    389 ms
  p99:                    571 ms
  Throughput:           39.80 req/sec

WHY IT LOOKS THIS WAY:
The load test was designed to stay BELOW the application's max limit (~59 req/sec).
At 50 users/sec the server handled every single request successfully.

What happened in the system:
- The AWS EC2 Tomcat instance had enough threads free to process every request
  without queuing. The 50 concurrent users stayed well within the 200-thread
  default Tomcat pool.
- Response times were extremely stable: p50=359ms, p95=389ms - a difference of
  only 30ms. This flat curve means the server was NOT under stress.
- The 320ms baseline is the real-world network round-trip time to AWS (internet
  latency Washington DC region), confirmed by the HAR recording (200ms TCP
  connect + 179ms server wait = ~396ms).
- The occasional spike to 1,858ms (max) is a normal JVM "hiccup" - the Java
  Garbage Collector paused briefly during the test. This is expected behavior
  in any Java application under sustained load.
- CONCLUSION: 50 req/sec is a comfortable operating point - the app is healthy,
  stable, and responsive. This is "normal working conditions."

TEST 2: STRESS TEST (5 minutes, ramp to 200 users/sec)
=====================================================================

RESULTS:
  Total requests:       30,615
  OK:                   14,424  (47% success)
  KO:                   16,191  (53% failure)
  Mean response time (OK):  1,477 ms
  Min:                    326 ms
  Max (OK):            12,489 ms
  p50 (OK):             1,363 ms
  p99 (OK):             7,422 ms
  Throughput (total):  101.04 req/sec
  Throughput (OK):      47.60 req/sec

WHY IT LOOKS THIS WAY:
The stress test deliberately pushed past the breaking point (ramp to 200 req/sec).
At around 60-70 users/sec the system started degrading. Above that it collapsed.

What happened in the system:
- PHASE 1 (warm-up, 1-10 req/sec, 30 sec): All OK, fast responses ~330ms.
  Server threads were idle, requests processed instantly.

- PHASE 2 (ramp 10->50 req/sec, 1 min): Mostly OK. Thread pool starts filling up.
  Response times begin climbing as threads queue up slightly.

- PHASE 3 (ramp 50->100 req/sec, 1 min): BREAKING POINT. Thread pool saturates.
  Tomcat has 200 threads but each request holds a thread for ~360-400ms, meaning
  max sustainable throughput = 200 threads / 0.4 sec = ~500 req/sec theoretical.
  However, AWS t2.micro CPU (1 vCPU) cannot process this many JSP renders.
  CPU becomes 100% saturated. Requests queue, waiting times explode.
  Gatling 10-second timeouts start firing -> KO cascade begins.

- PHASE 4 (ramp 100->200 req/sec, 1 min): Full overload. New requests arrive
  faster than old ones complete. Queue grows infinitely. 53% of all requests
  timed out. The server did NOT crash - it kept responding to queued requests,
  just too slowly.

- PHASE 5 (hold at 200 req/sec, 1 min): Sustained overload confirmed.
  The KO rate stabilizes at ~53% - the server can only serve ~47 req/sec
  successfully under these conditions (degraded performance mode).

- The one OK request that reached 12,489ms (above 10s timeout) shows that some
  connections were held open even past the Gatling cutoff - Tomcat was still
  trying to serve them but too slowly.

TEST 3: MAX LIMIT TEST (find the breaking point, ramp to 300 users/sec)
=====================================================================

RESULTS:
  Total requests:       43,545
  OK:                   16,322  (37% success)
  KO:                   27,223  (63% failure)
  Mean response time (OK):  1,164 ms
  Min:                    326 ms
  Max (OK):             8,049 ms
  p50 (OK):               478 ms
  p99 (OK):             7,462 ms
  Throughput (total):   157.77 req/sec
  Throughput (OK):       59.14 req/sec  <- THIS IS THE MAX LIMIT

WHY IT LOOKS THIS WAY:
This test was designed to find the exact breaking point by exceeding it significantly.
The max injection rate of 300 req/sec was 5x the max limit, deliberately chosen
to guarantee the system would break and reveal its true ceiling.

What happened in the system:
- PHASE 1 (1->50 req/sec over 1 min): Server handled everything fine.
  p50 response ~350ms, 0% failure. This is below the limit.

- PHASE 2 (50->150 req/sec over 1 min): Crossed the limit (~59 req/sec).
  Failures begin. The queue inside Tomcat's acceptor starts growing.
  Each new request waits longer for a thread. The 10-second clock starts ticking.
  By the end of this phase the majority of requests are timing out.

- PHASE 3 (150->300 req/sec over 1 min): Massive overload. For every 1 request
  the server can process successfully, 1.7 requests time out.
  The server is now in "survival mode" - it processes what it can but is
  overwhelmed by the injection rate.

- PHASE 4 (hold 300 req/sec for 1 min): Sustained 63% failure rate confirms
  the server's max sustainable throughput ceiling: 59 req/sec.

- PHASE 5 (cool down to 1 req/sec, 30 sec): The system recovers. Queued
  requests are processed or dropped. Response times return to baseline ~330ms.

- The bimodal distribution of response times tells the full story:
    - Successful requests: mean 1,164ms (slightly elevated but functional)
    - Failed requests: exactly 10,000ms (pure timeout, not server error)
  This confirms the server was QUEUING (not crashing) - a healthy but overloaded system.

- MAX LIMIT CONCLUSION: 59 req/sec is the ceiling because:
  1. AWS t2.micro has 1 vCPU - the CPU saturates around 59 concurrent JSP renders/sec
  2. Each JSP render requires reading/compiling the servlet, allocating memory,
     and writing the HTTP response - CPU-bound work
  3. The Tomcat thread pool is not the bottleneck (200 threads > 59 req/sec * 0.4s = 24 threads)
  4. The real bottleneck is CPU compute power of the free-tier EC2 instance
